DostLang Compiler and Interpreter — Implementation Plan
A complete, production-quality compiler construction project for DostLang, a custom Urdu/Hindi-inspired programming language.
Project Overview
DostLang demonstrates all classical phases of compiler construction: Lexical Analysis → Syntax Analysis → AST Generation → Semantic Analysis → Symbol Table → IR Generation → Interpretation/Execution
Tech Stack: Python 3.12+, PLY (lex-yacc), Tkinter, Graphviz, SQLite
Proposed Changes
1. Core Language Infrastructure
[NEW] 
ast_nodes.py
AST node classes for the entire language:
ProgramNode — wraps body between shuru/khatam
VarDeclNode — rakho x = expr;
AssignNode — x = expr;
BinOpNode — arithmetic & comparison (+, -, *, /, %, ^, ==, !=, >, <, >=, <=)
UnaryOpNode — unary minus, logical not
NumberNode, StringNode, BoolNode, IdentifierNode — literals & identifiers
PrintNode — dikhao(expr)
InputNode — pocho name;
IfNode — agar/warna with optional else
WhileNode — jabtak cond { body }
FuncDefNode — kaam name(params) { body }
FuncCallNode — name(args)
ReturnNode — wapas expr;
ArrayDeclNode, ArrayAccessNode — array support (extra feature)
ForNode — for loop support (extra feature)
CommentNode — single & multi-line comments
Each node stores line and col for error reporting.
[NEW] 
lexer.py
PLY-based lexical analyzer. Token types:
CategoryTokens
KeywordsSHURU, KHATAM, RAKHO, DIKHAO, POCHO, AGAR, WARNA, JABTAK, KAAM, WAPAS, SACH, JHOOT
LiteralsNUMBER_INT, NUMBER_FLOAT, STRING
IdentifiersID
ArithmeticPLUS, MINUS, TIMES, DIVIDE, MODULO, POWER
ComparisonEQ, NEQ, GT, LT, GTE, LTE
AssignmentASSIGN
DelimitersLPAREN, RPAREN, LBRACE, RBRACE, LBRACKET, RBRACKET, SEMICOLON, COMMA
LogicalAND, OR, NOT
Handles single-line (#) and multi-line (### ... ###) comments. Tracks line numbers and column positions. Reports lexical errors with position info.
[NEW] 
parser.py
PLY yacc-based parser. Grammar rules (simplified BNF):
program        → SHURU stmt_list KHATAM
stmt_list      → stmt stmt_list | ε
stmt           → var_decl | assignment | print_stmt | input_stmt
               | if_stmt | while_stmt | func_def | return_stmt | func_call SEMICOLON | expr SEMICOLON
var_decl       → RAKHO ID ASSIGN expr SEMICOLON
assignment     → ID ASSIGN expr SEMICOLON
print_stmt     → DIKHAO LPAREN expr_list RPAREN SEMICOLON
input_stmt     → POCHO ID SEMICOLON
if_stmt        → AGAR expr LBRACE stmt_list RBRACE (WARNA LBRACE stmt_list RBRACE)?
while_stmt     → JABTAK expr LBRACE stmt_list RBRACE
func_def       → KAAM ID LPAREN param_list RPAREN LBRACE stmt_list RBRACE
func_call      → ID LPAREN arg_list RPAREN
return_stmt    → WAPAS expr SEMICOLON
expr           → expr binop expr | unaryop expr | LPAREN expr RPAREN
               | NUMBER | STRING | SACH | JHOOT | ID | func_call | array_access
Operator precedence: ^ > * / % > + - > comparisons > logical. Error recovery with p_error() providing meaningful messages.
[NEW] 
symbol_table.py
Scope-aware symbol table with fields:
FieldDescription
nameVariable/function name
typeint, float, string, bool, function
scopeScope name (e.g., global, add, if_1)
valueCurrent value
lineDeclaration line number
Features:
Nested scope stack (push/pop)
declare(), lookup(), update(), get_all() methods
Scope chain resolution (inner → outer lookup)
[NEW] 
semantic.py
Semantic analysis pass over the AST:
Variable declaration checking — flag undeclared variables
Duplicate variable detection — flag re-declarations in the same scope
Type checking — validate operations (e.g., no string + int)
Function existence checking — flag calls to undefined functions
Arity checking — verify argument counts match parameter counts
Scope management — enter/exit scopes for functions, if, while
Return statement validation — ensure returns are inside functions
Collects all errors as a list with line numbers
[NEW] 
ir_generator.py
Three-address code (TAC) generator:
Generates temporaries t1, t2, ... for intermediate results
Labels L1, L2, ... for control flow
Operations: t1 = a + b, IF t1 GOTO L2, GOTO L3, PARAM x, CALL func, RETURN val
Outputs human-readable TAC listing
[NEW] 
interpreter.py
Tree-walking interpreter that directly executes the AST:
Separate environment stack from symbol table (runtime vs. compile-time)
Variable storage with nested scopes
Arithmetic evaluation with type coercion
Function calls with parameter binding and local scope
Loop execution with break detection
Input handling (reads from stdin or GUI input)
Runtime error handling with clear messages
Output capture for GUI display
[NEW] 
visualizer.py
AST and Parse Tree visualization using Graphviz:
generate_ast_graph(ast) → DOT → PNG/SVG
generate_parse_tree(ast) → DOT → PNG/SVG
Each node labeled with type and value
Color-coded by node type (statements=blue, expressions=green, literals=orange, etc.)
Saves to outputs/ directory
[NEW] 
database.py
SQLite database for execution history:
sql
CREATE TABLE executions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_code TEXT,
    output TEXT,
    tokens TEXT,
    ir_code TEXT,
    status TEXT,
    timestamp DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE TABLE errors (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    execution_id INTEGER,
    error_type TEXT,
    message TEXT,
    line_number INTEGER,
    timestamp DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (execution_id) REFERENCES executions(id)
);

Now I'm seeing a potential shift conflict between assignment statements and expressions that start with an identifier, since both can begin with ID. When the parser encounters ID with ASSIGN as the lookahead, it can shift to continue building an assignment or reduce ID to expr, but since ASSIGN can't follow expr in any rule, reducing would cause an error—so the parser shifts. For ID LPAREN, the parser recognizes this as the beginning of a function call and shifts accordingly, eventually reducing the complete func_call expression.

The if-else grammar rule is straightforward: agar followed by an expression and a block, optionally followed by warna and another block. Since warna is always followed by braces rather than another agar, there's no dangling-else ambiguity to worry about.
